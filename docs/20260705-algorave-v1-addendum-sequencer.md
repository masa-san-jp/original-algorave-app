# Algorave Terminals v1 — 追補: STEPモード(視覚的ステップシーケンサー)+ ループ進行プレイヘッド

作成日: 2026-07-05
版数: 1.0(機能仕様書 `20260703-algorave-v1-external-spec.md` への追補。既存FRを変更しない加算のみ)

## 背景

v1のコードエディタ(CODEモード)はStrudel/Tidal互換の全記法を表現できる反面、初見のユーザーにはテキスト入力の学習コストがある。「より視覚的・直感的に操作できるシーケンサーモード」と「ターミナル上でループが進行するUI」を追加要件として実装した。

## FR-13: STEPモード(ステップシーケンサー)

1. 各トラックのヘッダに `⊞ STEP` / `⌨ CODE` の切替ボタンを追加する。既定は CODE モード(既存動作は完全に維持)。
2. STEPモードは **BD / SD / HH / OH / CP / BS(ベース) / PD(シンセパッド) / SM(サンプラー) の8レーン × 16ステップ**の固定グリッドを表示する(1グリッド = 1小節、16分音符相当)。4ステップごとに区切り線を入れ拍を視認できるようにする。
3. セルをクリックすると即座にそのレーン・ステップがトグルされ、**次のサイクル境界から音に反映される**(RUNボタンを待たない)。これは既存の「AIはRUNを待つ(事故防止)」原則とは異なる意図的な設計判断であり、理由は次の通り:
   - STEPモードのセル操作は常に構文的に妥当な状態しか生成しない(パースエラーが原理的に起こり得ない)ため、CODEモードの「編集途中の壊れた構文」という事故リスクが存在しない。
   - 物理的なステップシーケンサー(TR-808等)の操作感は「叩いたら鳴る」が期待値であり、直感性を優先した。
4. **グリッドはコードの別ビュー**であり、独立した状態を持たない。セル操作は内部で `bd ~ ~ bd ...`(16トークン/行)形式のコードを生成し、既存の `parse()` → `runTrack()` パイプラインをそのまま通す(パーサ・スケジューラは無変更)。そのためCODEモードに戻すといつでも等価なコードが編集用テキストとして見える。
5. CODEモード→STEPモードへの切替時、現在の `code` を可能な範囲でグリッドへ変換する(`stepgrid.ts: codeToGrid`)。変換規則: 各行がちょうど16個の空白区切りトークンで、休符 `~` 以外はすべて単一レーン名(別名可: `sn`→SD, `kick`→BD, `clap`→CP, `bass`→BS, `pad`→PD, `sample`/`smp`→SM)のときのみ、その行を対応レーンへ反映する。1行でもこの形に一致しない場合(グループ・修飾子・音名・16以外の分割数等)は「現在のコードはSTEPモードで完全には表現できません(変換できた範囲のみ反映)」と表示し、変換できなかった分は空のまま新規グリッドとして開始する。
6. STEPモードは全記法のうち「フラットな等分割パターン」のみを表現する部分集合であり、`[ ]` `< >` ユークリッド等の高度な記法はCODEモードでのみ編集可能(既存のまま)。この非対称は意図的なスコープ:STEPモードは入門・直感操作、CODEモードは全記法という役割分担。

## FR-15: ベース(BS)・シンセパッド(PD)

1. `bs`(別名 `bass`)・`pd`(別名 `pad`)は、ドラムと同じ**固定ピッチのON/OFFトリガー**として扱う(ユーザー選定: 「固定ピッチのON/OFF」)。ステップごとの音程指定はv1では行わない。
2. `bs`: 固定ノート C1 を鳴らす単音シンセ(`Tone.MonoSynth`、サブ的な低域+短めのフィルタエンベロープ)。
3. `pd`: 固定コード(Cメジャートライアド `C3/E3/G3`)を鳴らすスロー・アタックのポリシンセ(`Tone.PolySynth` + fatsawtooth、attack 0.4s/release 1.6s)。1回のトリガーが小節を跨いで鳴り続けることでパッドらしい持続感を出す(スケジューラ自体は変更していない)。
4. 記法上は他のドラムトークンと同列の「サウンドトークン」として扱われ、CODEモードでも `bs ~ ~ bs` のように直接書ける。

## FR-16: アルペジエイター(ARP)

1. STEPモードに ARP 行を追加する。ルート音(C〜B)・オクターブ(2〜4)・コード種類(Major/Minor/Maj7/Min7/Dom7/Sus4)・パターン(UP/DOWN/UP-DOWN/RANDOM)・ON/OFFトグルで構成する(ユーザー選定: 「コード指定で自動生成」)。
2. 有効化・設定変更の都度、`arpeggiator.ts: arpToCodeLine()` が指定コードを分散和音化した**既存記法そのままの音名トークン16個**(例: `c3 e3 g3 c3 e3 g3 ...`)を生成し、コードの1行として合成する。**新しい記法・パーサ変更は一切ない**(既存のミニノーテーションで完全に表現できるため)。
3. RANDOMパターンは再現性のため擬似乱数(`Math.random`不使用、決定的なインデックス計算)で選択する。
4. CODEモードへ切り替えても生成済みの行はそのまま見える。ただしARP設定自体(ルート/コード種類等)はCODEモードのテキストから逆算しない(コード→ARP設定への一般的な逆変換は行わない、既知の制約)。

## FR-17: サンプラー(SM)

1. `sm`(別名 `sample`/`smp`)は1トラックにつき1スロットの音声を1ショット再生するトークン。未読込時は無音(パースは通るが発音しない)。
2. STEPモードに音声読み込み用のミニコントロールを追加する(ユーザー選定: 「アップロード、または内部で録音したファイルを取り込める」の両対応):
   - **📁 読込**: `<input type="file" accept="audio/*">` でローカルの音声ファイルを選択する。
   - **⏺ 録音**: `getUserMedia` + `MediaRecorder` でマイクから直接録音し、停止すると自動的にサンプルとして読み込まれる。
   - 両方とも内部的には同じ経路(`Blob` → `URL.createObjectURL` → `Tone.Player.load(url)`)で統一しており、読み込み後は即座に objectURL を revoke してリークを防ぐ。
3. 音声バッファは `Instrument`(audio.ts)が保持する実行時状態であり、**localStorageには永続化しない**(既存の FR-11 の対象外。理由: 音声バイナリの永続化はサイズ制約・IndexedDB設計を要し、v1のスコープでは見送り、既知の制約として明記する)。リロードすると読込状態はリセットされる。
4. マイク権限が得られない・非対応ブラウザの場合は演奏を止めずに `t.error` へエラー文言を表示する(既存のエラー表示規約に準拠)。

## FR-14: ループ進行プレイヘッド

1. **CODEモード**: RUNNING中のトラックは、エディタ領域の左端から右端まで1小節(`(60/BPM)×4`秒)かけて走査する縦の光る線(プレイヘッド)を表示する。既存の枠発光(`beatPulse`)と同じ「`.running`クラス + `var(--measure)`によるCSSアニメーション」の規約に従うため、BPM変更に自動追従する。
2. **STEPモード**: RUNNING中のトラックは、16ステップを離散的に(`steps(16, jump-start)`)チェイスするハイライトを表示し、現在鳴っているステップ位置を直接示す。
3. 両方とも `prefers-reduced-motion: reduce` で無効化される(既存のNFR-07に準拠)。

## スコープ外・既知の制約

- STEPモードの表示/非表示(`mode`)・グリッド内容(`grid`)・ARP設定(`arp`)・サンプラーの音声バッファはlocalStorageに永続化しない。リロード後は常にCODEモードに戻るが、演奏内容そのもの(`code`。ARP/グリッドが生成した行を含む)は既存のFR-11でそのまま復元される(音声バッファ以外の音楽的内容は失われない)。
- ステップ数は16固定(v1では可変不可)。8/32等の解像度切替は将来検討。
- STEPモードのセル操作による即時反映(RUNを待たない)は、CODEモードのテキスト編集には一切影響しない(CODEモードは既存どおりRUN必須のまま)。
- ベース/パッドは固定ピッチ・固定コードのみ(ステップごとの音程指定は非対応)。サンプラーは1トラック1スロットのみ(複数サンプルのキットは非対応)。いずれも将来の拡張候補。
