# Algorave Terminals v1 — 機能仕様書(発注用)

作成日: 2026-07-03
版数: 1.0
ステータス: 発注可能(本書・記法仕様書・受け入れテスト仕様書・実装計画書の4点で構成)

## 1. 目的と全体像

ブラウザだけで動くライブコーディング音楽アプリ「Algorave Terminals」の v1 を実装する。
トラック = ターミナルウィンドウというUIで、コードを書いて RUN すると次の小節から音が変わる。デスクトップでは複数ターミナルを並べてミキシングし、スマホでは TRACK A/B をタブで行き来して演奏する。

コード記法は **Strudel / TidalCycles のミニノーテーション互換サブセット**とする(詳細は `20260703-algorave-v1-notation-spec.md`)。既存のライブコーディング人口がそのまま乗り換えられることを製品要件とする。

スコープ外の要望が発生した場合は変更管理プロセス(実装計画書 §6)に従う。

## 2. 用語集

| 用語 | 定義 |
|---|---|
| サイクル | 1小節(4/4拍子)。すべてのパターンは1サイクルを周期とする |
| トラック | 1つのターミナルウィンドウ。独立したパターン・音源・ミキサーを持つ |
| RUN | エディタの内容をパースし、**次のサイクル境界から**再生パターンに反映する操作 |
| ステージング | AI生成コードはエディタに挿入するだけで、人間が RUN するまで音に反映しない方式 |
| イベント | パターンをパースして得られる発音1回分。`{pos, dur, tok}`(posはサイクル内位置 0≦pos<1) |
| ゴールデンテストベクタ | 記法仕様書 §8 の入力→期待イベント列の表。単体テストとして全件実装・全件パスが検収条件 |

## 3. アーキテクチャ要件

### 3.1 モジュール構成

実装は以下のモジュールに分離すること(ファイル分割の単位は任意だが、依存方向を守ること):

```
parser   … テキスト → AST。純粋関数。DOM・Tone.js に依存しない
pattern  … AST + cycleIndex → イベント列(query関数)。純粋関数
audio    … Tone.js ラッパ。音源生成・トリガー・ミキサー・トランスポート
scheduler… サイクルごとに pattern.query を呼び、audio に発音予約する
ui       … DOM構築・イベントハンドラ・状態描画
ai       … Anthropic API 呼び出し
storage  … localStorage 永続化
```

- `parser` と `pattern` は**ブラウザAPI非依存の純粋モジュール**とし、Node 上で単体テスト可能にすること(検収条件)。
- パターンは静的イベント列ではなく `query(cycleIndex) → Event[]` として評価すること。`<a b>`(サイクル交替)・`?`(確率)・`/n`(減速)がサイクル番号に依存するため。

### 3.2 技術スタック(拘束条件)

| 項目 | 指定 |
|---|---|
| 言語 | TypeScript(strict: true) |
| ビルド | Vite。成果物は `dist/` の静的ファイル一式(サーバーサイド処理なし) |
| 音声 | Tone.js ^14.8(npm 依存としてバンドルする。CDN読み込み禁止 = 読込後はオフラインで音が出ること) |
| UIフレームワーク | 使用しない(Vanilla TS + DOM)。※軽量な状態管理の自作は可 |
| テスト | Vitest。parser / pattern の単体テスト必須 |
| Lint/Format | ESLint + Prettier(設定ファイルを納品物に含める) |
| 対応配信 | `python3 -m http.server` 等の単純な静的サーバーで `dist/` がそのまま動くこと。サブパス配信も考慮し、アセット参照は相対パスとする |

## 4. 機能要件

各要件にIDを付す。受け入れテスト仕様書のテストケースはこのIDに対応する。

### FR-01 起動オーバーレイ(オーディオアンロック)

1. 初回表示時、全画面オーバーレイを表示する。内容: ロゴ、「▶ タップして演奏開始」ボタン、遊び方3ステップ、無音時の注意書き(文言はプロトタイプと同一)。
2. ボタンタップで `Tone.start()` を呼び(iOS Safari のユーザー操作起点要件)、オーバーレイを閉じ、全トラックのデモパターンを RUN 状態で再生開始する。
3. オーバーレイ表示中に個別トラックの RUN が押された場合も同様にアンロックしてオーバーレイを閉じる。

### FR-02 トラック管理

1. 初期状態はトラック2本。デモパターンは以下(プロトタイプと同一):
   - TRACK A: `bd ~ bd ~` / `~ sn ~ sn` / `hh*8`(3行)
   - TRACK B: `c2 ~ [c2 c3] g1` / `~ ~ eb2 ~`(2行)
2. `+ TRACK` で末尾に追加(初期コード `bd ~ ~ bd`)。上限6本。上限時はボタンを disabled にする。
3. トラック名は表示順で `TRACK A`〜`TRACK F` を自動割当(削除時に詰めて振り直す)。
4. 枠色はプロトタイプの HUES 配列 `#ffb454 #59f7ff #ff5ce1 #a6ff4d #c09bff #ff7a59` を表示順に適用する。
5. 各トラックのヘッダ右端の × で削除できる。最後の1本は削除不可(×を disabled)。削除時は当該トラックの音源ノードを dispose すること(メモリリーク禁止)。
6. モバイル表示(FR-09)では + TRACK と × を非表示にする。

### FR-03 エディタと RUN

1. 各トラックは複数行テキストエディタ(等幅フォント、スペルチェック無効)を持つ。
2. RUN の起動方法: ① RUN ボタン ② エディタフォーカス中の Ctrl+Enter / ⌘+Enter(デフォルト改行動作は抑止)。
3. RUN 時の動作:
   - エディタ全文をパースする。
   - **成功時**: 次のサイクル境界から新パターンに切り替える(現サイクルで既に発音予約済みのイベントはそのまま鳴らしてよい)。エラー表示をクリアする。
   - **パースエラー時**: 現在再生中のパターンを**維持したまま**、エラー行の下部にエラーメッセージ(記法仕様書 §7 の固定文言)を表示する。演奏中に音が止まらないことがライブ用途の必須要件。
   - **有効イベント0件(空・コメントのみ)時**: 当該トラックを次サイクルから停止し、メッセージ「有効なトークンがありません」を表示する。
4. RUN してもエディタ内容・カーソル位置は変化しない。

### FR-04 再生制御

1. マスター `▶ START ALL` : オーディオアンロック後、全トラックをパースし、有効なパターンを持つ全トラックを再生開始。トランスポートを開始する。再生中はボタンが `■ STOP ALL`(色 `#ff5ce1`)に変わる。
2. `■ STOP ALL` : トランスポートを停止し、全トラックを IDLE にする。サイクルカウンタは停止(リセットはしない)。
3. トラック `STOP` : 当該トラックのみ次サイクルから停止。他トラックは継続。IDLE 中は disabled。
4. トラック `MUTE` / `UNMUTE` : 発音を止める(パターン評価は継続してもよい)。MUTE 中はヘッダに `MUTED`(色 `#ff5ce1`)を表示し、ボタン表記を `UNMUTE` に変える。MUTE はトラックの RUNNING 状態を変えない。
5. 状態表示: ヘッダに `RUNNING` / `IDLE` を表示する。

### FR-05 BPM

1. ヘッダのスライダーで 80〜180 BPM(整数、初期値 132)。現在値を数字で併記。
2. 変更は即時反映(再生中でも切れ目なく)。サイクル長・発光アニメーション周期(FR-08)も追従する。

### FR-06 トラックミキサー

1. `VOL` スライダー: 0.00〜1.00(step 0.01、初期値 0.80)。トラックゲインに 50ms のランプで反映。
2. `FILT` スライダー: 0〜100(初期値 100)。ローパスフィルタのカットオフ周波数を次式で設定し、50ms のランプで反映する:
   `freq = 80 × 150^(x/100)` (x=0 → 80Hz、x=100 → 12kHz)
3. どちらも再生中にリアルタイムで効くこと(ザッピングノイズが知覚されないこと)。

### FR-07 記法(パーサ)

記法仕様書 `20260703-algorave-v1-notation-spec.md` の全項目。同書 §8 のゴールデンテストベクタを単体テストとして全件実装し、全件パスすること。

### FR-08 サイクル表示・ビート同期発光

1. ヘッダに現在のサイクル番号を表示する(トランスポート開始からの累計。STOP ALL でも維持、ページリロードで 0)。
2. RUNNING かつ非MUTE のトラックは、枠の発光がサイクル周期(= 1小節 = `(60/BPM)×4` 秒)で明滅する(プロトタイプの `beatPulse` と同等)。
3. `prefers-reduced-motion: reduce` 環境ではアニメーションを止め、静的な発光に置き換える。

### FR-09 モバイル(幅760px未満)

1. ビューポート幅 760px 未満で以下に切り替わる(リサイズ時も動的に追従):
   - `TRACK A` / `TRACK B` の大型タブを表示し、選択中の1トラックのみ表示する。
   - 再生中のタブにはラベル末尾に ` ●` を付す。
   - 3本目以降のトラックは非表示(タブも出さない)。
   - `+ TRACK` とトラック削除 × は非表示。
2. 期待ユースケース: 片方を鳴らしたままもう片方のタブでコードを書き RUN する A/B 奏法が途切れなくできること。

### FR-10 AI 生成(ステージング方式)

1. 画面上部に「AI指示」テキスト欄と「APIキー」パスワード欄を置く(プレースホルダ文言はプロトタイプと同一)。
2. 各トラックの `✦ AI` ボタンで Anthropic Messages API を呼ぶ(§9 参照)。
3. 成功時: 生成パターンを**エディタに挿入するだけ**(自動RUNしない)。メッセージ「AI案を挿入しました。RUNで反映」を表示。挿入前の再生状態は変化しない。
4. 失敗時: 再生を止めず、エラー文言(§10)を表示する。
5. リクエスト中は当該ボタンを disabled + 表記 `AI…` にする。多重送信禁止。30秒でタイムアウトしエラー扱いとする。
6. APIキー未入力で押した場合: 「AI機能にはAPIキーが必要です(上の入力欄へ)」を表示し、リクエストしない。

### FR-11 永続化(localStorage)

1. APIキー: `✦ AI` 実行時に `algorave_api_key` キーへ保存。起動時に復元して入力欄に反映。
2. パターン自動保存: 各トラックのエディタ内容・トラック数を、入力からデバウンス1秒で `algorave_patterns_v1`(JSON)へ保存。起動時に存在すれば復元し、無ければ FR-02 のデモパターンを使う。BPM も同様に保存・復元する。
3. localStorage が使えない環境(プライベートモード等)では例外を握りつぶし、保存なしで動作継続すること。

### FR-12 ヘルプ表示

画面下部に記法チートシートを常時表示する。v1 の記法(ユークリッド `(3,8)`、交替 `<a b>`、確率 `?`、`!` `@` `/` `,` を含む)を反映した内容に更新すること。1画面に収まる密度を維持する(詳細は記法仕様書へのリンクで足りる)。

## 6. 音源仕様

全トラック共通で以下の構成を、トラック生成時に個別インスタンス化する:

```
[各音源] → Gain(トラックVOL, 初期0.8) → Filter(lowpass, 初期12000Hz) → Master Gain(0.9) → destination
※ hh / oh のみ 8kHz highpass を経由して Gain へ
```

| トークン | Tone.js 音源 | パラメータ | trigger |
|---|---|---|---|
| `bd`(別名 `kick`) | MembraneSynth | octaves:6, pitchDecay:0.045 | "C1", "8n" |
| `sd`(別名 `sn`) | NoiseSynth | pink, env{attack:0.001, decay:0.14, sustain:0} | "16n" |
| `hh` | NoiseSynth | white, env{attack:0.001, decay:0.04, sustain:0} | "32n" |
| `oh` | NoiseSynth | white, env{attack:0.001, decay:0.25, sustain:0} | "16n" |
| `cp`(別名 `clap`) | NoiseSynth | pink, env{attack:0.002, decay:0.2, sustain:0} | "16n" |
| 音名(`c2` `g#3` `eb2` 等) | PolySynth(Synth) | sawtooth, env{attack:0.005, decay:0.15, sustain:0.15, release:0.08}, volume:-8dB | 音名を大文字化, "16n" |

- trigger 時の例外は捕捉し、演奏を止めないこと(高速連打での Tone.js 例外対策)。
- トラック削除時は全ノードを dispose する。

## 7. スケジューラ仕様

1. `Tone.Transport.scheduleRepeat(callback, "1m")` で1小節ごとにコールバックを受け、各 RUNNING トラックについて `pattern.query(cycleIndex)` を評価し、`time + pos × 小節秒` で発音予約する。
2. サイクルカウンタはこのコールバックで +1 し、`Tone.Draw` 経由で表示更新する(オーディオスレッド基準の表示同期)。
3. RUN による差し替え・STOP は「次のコールバックから反映」で実現する(FR-03/04 の次サイクル反映)。
4. BPM 変更は `Tone.Transport.bpm.value` へ即時反映。
5. トランスポート開始は `Tone.start()` 解決後に `+0.05` オフセットで行う。

## 8. 状態モデル

```
AppState  = { playing: bool, cycle: int, bpm: int, mobileTab: 0|1, booted: bool }
Track     = { id, code: string, ast: AST|null, running: bool, muted: bool,
              gain: number, cutoff: number, error: string|null }
```

- `running && playing && !muted` のときのみ発光(FR-08)・発音する。
- 全トラックの状態遷移(IDLE ⇄ RUNNING、MUTE トグル)は上記フラグの組合せで表現でき、UIは状態から一意に描画されること(描画関数の冪等性)。

## 9. AI 連携仕様

| 項目 | 値 |
|---|---|
| エンドポイント | `POST https://api.anthropic.com/v1/messages` |
| ヘッダ | `Content-Type: application/json`, `x-api-key: <入力値>`, `anthropic-version: 2023-06-01`, `anthropic-dangerous-direct-browser-access: true` |
| model | `claude-sonnet-4-6`(ソース内の単一定数として定義し、差し替え可能にする) |
| max_tokens | 1000 |
| timeout | 30秒(AbortController) |

- system プロンプトには v1 の記法一覧(ユークリッド・交替・確率・`!` `@` `/` `,` を含む)を明記し、応答形式を `{"pattern":"line1\nline2"}` の JSON のみと指示する。2〜5行・演奏可能・踊れるパターンという制約もプロトタイプ同様に含める。
- user メッセージには BPM・当該トラックの現在パターン・AI指示欄の内容(空なら既定指示「グルーヴを保ちつつ変化とエネルギーを加える」)に加え、**他トラックの現在パターンをコンテキストとして含める**(アンサンブル提案のため。既存仕様書「次の一手」#3 を v1 に取り込み)。
- 応答処理: `content` の text ブロックを連結 → ` ```json ` フェンスを除去 → JSON.parse → `pattern` キー必須。得られたパターンを**クライアント側でもパースし、エラーがあれば「AI案の記法にエラーがあります」として挿入はするがエラー表示を出す**(ユーザーが修正して RUN できるように)。

## 10. エラー処理一覧(文言は固定)

| # | 状況 | 表示(トラックのエラー行) | 再生への影響 |
|---|---|---|---|
| 1 | パースエラー | 記法仕様書 §7 の文言 | 旧パターン継続 |
| 2 | 有効トークン0件で RUN | `有効なトークンがありません` | 次サイクルから停止 |
| 3 | APIキー未入力で AI | `AI機能にはAPIキーが必要です(上の入力欄へ)` | なし |
| 4 | AI HTTP エラー | `AI生成に失敗: api {status}(キーとネット接続を確認)` | なし |
| 5 | AI タイムアウト/ネットワーク | `AI生成に失敗: {message}(キーとネット接続を確認)` | なし |
| 6 | AI 応答が不正(JSON/patternなし) | `AI生成に失敗: 応答を解釈できません` | なし |
| 7 | AI案に記法エラー | `AI案の記法にエラーがあります。修正してRUNしてください` | なし(挿入はする) |

## 11. 非機能要件

| ID | 要件 |
|---|---|
| NFR-01 | 対応ブラウザ: Chrome / Edge / Firefox / Safari の各最新2バージョン(デスクトップ)、iOS Safari 16.4+、Android Chrome 最新。上記で FR 全項目が動作すること |
| NFR-02 | RUN のパース+反映処理は 20行×16トークンのパターンで 10ms 以内(mid-tier ノートPC基準) |
| NFR-03 | 6トラック同時再生・BPM180 で音切れ・クリックノイズが知覚されないこと(デスクトップChrome基準) |
| NFR-04 | バンドルサイズ: Tone.js 込み gzip 350KB 以下 |
| NFR-05 | 初回ロード後はネットワーク断でも演奏機能が動作すること(AI機能を除く) |
| NFR-06 | XSS: §5.3 の制約を満たすこと(エディタに `<img src=x onerror=alert(1)>` を入れて RUN してもスクリプトが実行されない) |
| NFR-07 | アクセシビリティ: 全操作要素にアクセシブルネーム(aria-label 等)、キーボードのみで全機能操作可能、`prefers-reduced-motion` 対応 |
| NFR-08 | コンソールにエラー・警告を出さない(通常操作範囲で) |

## 12. 納品物
