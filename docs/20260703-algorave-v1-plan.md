# Algorave Terminals v1 — 実装計画書

作成日: 2026-07-03
版数: 1.0

## 1. 前提

- 契約範囲 = 機能仕様書・記法仕様書・受け入れテスト仕様書・本書の4点(§ 参照関係は機能仕様書 §1.1)。
- 参照プロトタイプ `index.html` はリポジトリに同梱済み。実装者は着手前に必ず動かして演奏感覚を確認すること。
- 想定規模: 実装 15〜20人日(1名、4〜5週間)。テスタ(実機検証)0.5人日×2回。
- 成果物は本リポジトリへの PR。マイルストーンごとに PR を分けてレビューを受ける(手戻り防止の中核施策)。

## 2. マイルストーンとゲート

各マイルストーンは「成果物」「完了条件(ゲート)」を満たして初めて次へ進む。ゲートは発注側レビュー(2営業日以内に回答)。

### M0: 環境構築(0.5週)

- 成果物: Vite + TypeScript(strict) + Vitest + ESLint/Prettier の雛形、GitHub Actions CI(lint/test/build)、空の `parser`/`pattern` モジュールと1本のダミーテスト。
- ゲート: CI グリーン。`npm run dev` / `npm run build` / `npm test` が README 記載どおり動く。

### M1: パーサ+パターン評価(1週)★最重要

- 成果物: `parse()` / `query()` の完全実装、ゴールデンテストベクタ GV 全件+Bjorklund 規範ベクタのテスト、エラー仕様(E-01〜E-06)の実装。
- ゲート: **ゴールデンテスト 100% パス、カバレッジ branch 90%+**。UI・音は不要。
- 狙い: 記法は本プロジェクトで唯一「解釈揺れ→手戻り」が起きうる領域。ここを純粋関数+自動テストで先に固定する。

### M2: オーディオエンジン+デスクトップUI(1.5週)

- 成果物: 音源(機能仕様書 §6)、スケジューラ(§7)、デスクトップUI一式(FR-01〜FR-08、FR-11、FR-12)。デモパターンが鳴り、ライブコーディングできる状態。
- ゲート: 受け入れテスト §3 の「起動・再生」「ライブコーディング」「ミキサー・BPM」「トラック管理」を ENV-1 で実施し全パス。デモ動画(1〜2分)提出。

### M3: モバイル+AI(1週)

- 成果物: FR-09(A/Bタブ)、FR-10(AI連携)、アクセシビリティ(NFR-07)。
- ゲート: 受け入れテスト §3 の「モバイル」「AI」を ENV-1/ENV-4(またはENV-5)で実施し全パス。

### M4: 検収(0.5〜1週)

- 成果物: 受け入れテスト全環境の実施結果表、P1 項目の実施、既知課題リスト、README 更新(ビルド・起動・記法チートシートへのリンク)。
- ゲート: 機能仕様書 §13 の検収条件を満たす。

## 4. リスクと対策

| リスク | 対策 |
|---|---|
| 記法の解釈違い(最大の手戻り要因) | ゴールデンテストベクタを契約に含めた。仕様の不明点は実装前に Q&A(§5)で解消し、**推測実装を禁止** |
| iOS Safari の音声制限・レイテンシ | プロトタイプで方式実証済み(タップ起点 `Tone.start()`)。M3 ゲートで実機必須 |
| Tone.js の高速連打例外・ノードリーク | trigger の例外捕捉を仕様化済み(§6)。T-70/T-71 で検出 |
| AI API の CORS・キー無効 | `anthropic-dangerous-direct-browser-access` ヘッダで直叩き可(プロトタイプ実証済み)。エラー文言を仕様で固定済み |
| 浮動小数の位置誤差(1/3 分割等) | 許容誤差 1e-9 をテスト仕様に明記済み |
| スコープクリープ | §6 変更管理。スコープ外リストを機能仕様書 §1.3 に明記済み |

## 5. コミュニケーション・Q&A

- 仕様の疑問は GitHub Issue(ラベル `question`)で起票。発注側は2営業日以内に回答し、回答は該当仕様書に追記して版数を上げる(口頭・チャット限りの仕様変更は無効)。
- 週1回、進捗と翌週予定を Issue で報告。
- 仕様書間に矛盾を見つけた場合: 記法仕様書 > 機能仕様書 > 参照プロトタイプ の優先順で解釈し、必ず Issue で確認する。

## 6. 変更管理

1. 仕様変更の要望は Issue(ラベル `change-request`)で起票し、影響範囲(工数・スケジュール)を見積もってから双方合意。
2. 合意済み変更は仕様書に反映し版数を上げてから実装する。
3. スコープ外項目(機能仕様書 §1.3)は v2 バックログとして記録のみ行う。
