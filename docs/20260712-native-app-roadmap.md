# Algorave Terminals — ネイティブアプリ化ロードマップ

作成日: 2026-07-12
版数: 1.0(方針文書。確定仕様ではなく、意思決定のための比較・計画)

## 目的

「まず Web(HTML)版でユーザー体験(UX)を詰め、十分になったらネイティブアプリ(iOS/Android)としてストア公開する」という2フェーズ戦略の、現実的な進め方・技術選択・チェックリストをまとめる。

## 大前提: 「Swift に変換」は自動ではできない

Web Audio(Tone.js)+ DOM で作った本アプリを、Swift ネイティブへ機械的に変換するツールは存在しない。アプリ化には方式選択が必要で、トレードオフが大きく異なる。

## アプリ化の3方式

| 方式 | 中身 | 既存Web資産の活用 | 音のレイテンシ | 工数 | ストア公開 | 主なリスク |
|---|---|---|---|---|---|---|
| **A. WebViewラップ**(Capacitor / Tauri v2 / 素のWKWebView) | Swift/Kotlin は薄いシェル。中身は今のWeb版をWebViewで内包 | ほぼ100% | iOS WKWebView の Web Audio は実用十分(プロの極限用途を除く) | 小 | ◎ App Store / Google Play 可 | WebView 特有の描画/オーディオ制約、審査で「ただのWebラッパー」と見なされない工夫が要る |
| **B. ネイティブ再実装**(Swift + AVAudioEngine 等) | 音源・スケジューラをネイティブ再実装。記法パーサ等の純ロジックは移植可 | 純ロジックのみ(parser/pattern/euclid/stepgrid/arpeggiator)。音響・UIは書き直し | 最良(サンプル精度のスケジューリング) | 大 | ◎ | 音響エンジン全書き直しの工数、iOS/Android 二重実装 |
| **C. PWA**(ストア非経由) | Web版に manifest + Service Worker。ホーム画面に追加 | 100% | 実用十分 | 極小 | △(iOS はプッシュ/バックグラウンド等に制約、ストア掲載なし) | iOS Safari の PWA 制約、発見性が低い |

### 推奨: A(WebViewラップ)を軸に

「HTMLで UX を詰めてから」という順序を選ぶなら、**磨いた Web版がそのままアプリの中身になる A が最適**。とくに Capacitor は iOS/Android 両対応・プラグイン充実・審査実績が豊富。B(全書き直し)は「Web Audio ではどうしてもレイテンシが足りない」とプロ用途で判明したときの最終手段。

本リポジトリは既に A に有利な資産を持つ:
- `npm run build:single` による**単一 HTML**(外部参照ゼロ)。WebView に同梱する中身としてそのまま使える。
- 純ロジックが `parser/pattern/euclid/stepgrid/arpeggiator` に分離済み。将来 B に進む場合も、ここは TypeScript のまま or Swift 移植の仕様として流用できる。
- `vite.config.ts` の `base: './'` 相対配信 → WebView のローカルファイル配信でパス問題が出ない。

## フェーズ1: Web版で UX を詰める(ネイティブ化前に潰す)

ネイティブ化しても WebView の中身は Web版なので、UX の作り込みはこのフェーズで完結させるのが効率的。チェックリスト:

### 1-1. オンボーディング・学習
- [ ] 初見で「記法」と「STEPモード」をどう伝えるか(現状はヘルプ1行)。インタラクティブな初回ガイド or サンプル読み込み
- [ ] STEP と CODE の行き来が直感的か。モード切替の発見性
- [ ] デモパターンのバリエーション(ジャンル別プリセット)

### 1-2. タッチ操作(スマホが主戦場)
- [ ] STEPセルの押しやすさ(ヒット領域・間隔)、連続入力のドラッグ(なぞって複数ON)
- [ ] 誤タップ・取り消しやすさ、片手操作の到達性
- [ ] スクロールとセル操作の競合(横スクロールするグリッドでの誤操作)
- [ ] ピンチ/ロングタップ等のジェスチャ余地

### 1-3. オーディオ・レイテンシ
- [ ] iOS 実機での発音遅延の実測(タップ→音、STEP反映)
- [ ] サイレント/着信/バックグラウンド復帰後の再開挙動
- [ ] AudioContext の suspend/resume ハンドリング(WebView でも重要)
- [ ] マスターのクリップ防止(トラック増加時の音割れ)

### 1-4. 表現力(元仕様「次の一手」より)
- [ ] ディレイ/リバーブのセンド、`hh*4?` 等の演奏表現
- [ ] 音色プリセット、STEPのベロシティ/確率
- [ ] パターンの書き出し/読み込み・共有(現状 localStorage のみ)

### 1-5. 仕上げ
- [ ] 空状態・エラー時の文言と見た目
- [ ] 縦横回転、各画面サイズ(小型スマホ〜タブレット)
- [ ] アクセシビリティ(コントラスト、reduced-motion は対応済み)

> フェーズ1の完了目安: 「実機で30分触っても操作に迷わず、音が破綻せず、作ったパターンを保存/再開できる」。

## フェーズ2: ネイティブ化して公開(方式 A 前提の流れ)

1. **技術検証(小さく早く)**: Capacitor で現行 Web版をラップし、iOS シミュレータ/実機で「起動→音が鳴る→STEP操作」まで通す。ここで WebView の Web Audio レイテンシと審査リスクを早期に見極める。
2. **ネイティブ体裁**: アプリアイコン、スプラッシュ、ステータスバー、セーフエリア、画面回転ロック、no-bounce スクロール。
3. **審査対策**: 「単なる Web ラッパー」で弾かれないよう、オフライン動作・ネイティブらしい体験(触覚フィードバック、共有シート等)を1つ以上入れる。
4. **配布準備**: Apple Developer Program(有料・年額)、App Store Connect 登録、プライバシー入力(このアプリは基本データ収集なし。マイク=サンプラー録音の用途説明が必要)、スクショ・説明文。Android は Google Play Console(初回登録料)。
5. **提出・審査・公開**。

### 音声まわりのネイティブ注意
- iOS: マイク使用(サンプラー録音)は `NSMicrophoneUsageDescription` が必須。使わないなら機能ごと落とす選択も。
- WebView 内 Web Audio はユーザー操作起点で AudioContext を resume する必要(現行の起動オーバーレイがそのまま活きる)。
- バックグラウンド再生を許すか(オーディオセッションのカテゴリ設定)は要件次第。

## 実装状況(このリポジトリ)

- **Capacitor 統合済み(方式A の土台)**: `app/capacitor.config.ts`(`webDir: dist` / `appId: com.algorave.terminals` / `appName: Algorave Terminals`)。ビルド済みの Web 版をそのままネイティブアプリに内包する。
- **手順(iOS の例。Mac + Xcode が必要)**:
  1. `cd app && npm install`
  2. `npm run cap:add:ios`(初回のみ。`ios/` を生成。Android は `npm run cap:add:android`)
  3. `npm run cap:ios`(`npm run build` → Web版を同梱 → Xcode が開く)
  4. Xcode で署名(Apple Developer アカウント)・アプリアイコン・表示名を設定 → **Archive** → **App Store Connect** へアップロード → 審査提出
- **ネイティブプロジェクト(`ios/` `android/`)はリポジトリに含めない**方針(`.gitignore` 済み)。各自の環境で `cap add` して生成する。CI で Web版の検証は継続し、ネイティブビルドは手元 or 専用CI(Mac ランナー)で行う。
- 残タスク: アプリアイコン/スプラッシュ、iOS の `NSMicrophoneUsageDescription`(サンプラー録音を使う場合)、審査対策(オフライン動作・触覚フィードバック等でネイティブ体験を1つ以上)。
